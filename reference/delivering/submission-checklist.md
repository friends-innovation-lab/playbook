# Submission Checklist

For government procurement challenge responses specifically.
Complete this checklist before submitting anything.

## Technical

- [ ] Live URL is working and stable
- [ ] URL is `[name].lab.cityfriends.tech` — not a `.vercel.app` URL
- [ ] All challenge requirements are addressed (re-read the brief)
- [ ] Primary user flow works end to end
- [ ] Auth works with demo credentials
- [ ] Data saves and persists correctly
- [ ] `npm run test:a11y` passes
- [ ] Lighthouse accessibility score 90+
- [ ] No console errors

## Content

- [ ] App name and description reflect the challenge context
- [ ] All copy is proofread — no typos, no lorem ipsum
- [ ] Agency name or context is visible where appropriate
- [ ] "Built by Friends Innovation Lab" is in the footer
- [ ] Demo credentials are documented somewhere reviewers can find them

## Documentation (if required by the challenge)

- [ ] One-pager written explaining the approach
- [ ] Technical approach addresses: stack choice, accessibility, data model
- [ ] Screen recordings or screenshots prepared as backup
- [ ] CLAUDE.md is up to date (for internal reference)

## Final check

- [ ] Someone who has NOT worked on the project has tested the full flow
- [ ] Team has rehearsed a 5-minute verbal walkthrough
- [ ] Backup plan is ready if the live URL fails during evaluation

## After submission

- [ ] Note the submission date in the project's CLAUDE.md
- [ ] Schedule a debrief for after results are announced
- [ ] If not selected: run the teardown script within 30 days
- [ ] If selected: update CLAUDE.md with next phase details
